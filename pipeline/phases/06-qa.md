## QA Process

### 1. Smoke Test

Can the app start without errors?
- Start the application
- Check for console errors
- Verify the main page/screen renders

If the app won't start, that's a critical bug. Log it and stop.

For web apps using browse:
```bash
$B goto "$APP_URL"
$B console --errors
$B snapshot -i -a -o "{PHASE_ARTIFACTS}/screenshots/initial.png"
```

### 2. Success Criteria Verification

Read the success criteria from `{PHASE_ARTIFACTS}/phase-01-plan-ceo.md`.
Test each criterion literally:

For each criterion:
- Attempt the exact user action described
- Record PASS or FAIL
- If FAIL: describe what happened vs. what should have happened
- Take a screenshot of the result

For web apps, use browse for evidence:
```bash
$B screenshot "{PHASE_ARTIFACTS}/screenshots/criterion-N.png"
$B snapshot -D   # diff to see what changed after an action
```

### 3. Edge Case Testing

For each user-facing feature:
- **Empty input:** What happens?
- **Invalid input:** What happens?
- **Rapid repeated actions:** What happens?
- **Browser back button:** What happens? (if web app)
- **Refresh during operation:** What happens? (if web app)

### 4. Visual Inspection (Web Apps)

If the app has a UI, use the browse binary:

```bash
$B goto "$APP_URL"
$B snapshot -a -o "{PHASE_ARTIFACTS}/screenshots/main-annotated.png"
$B screenshot "{PHASE_ARTIFACTS}/screenshots/main-page.png"
```

Check:
- Does it look broken? Overlapping elements, missing styles?
- Is text readable? Sufficient contrast?
- Does it work on a narrow viewport?
  ```bash
  $B viewport 375 812
  $B screenshot "{PHASE_ARTIFACTS}/screenshots/mobile.png"
  $B viewport 1280 720
  ```

### 5. Test Suite Verification

Run the test suite and record results:
```bash
cd output && npm test 2>&1
```

All tests should pass. A failing test is a bug.

### 6. Regression Test Generation

For each bug found during QA:
1. Write a regression test that reproduces the exact scenario.
2. The test MUST:
   - Set up the precondition that triggered the bug
   - Perform the action that exposed it
   - Assert the correct behavior (not just "it renders" or "it doesn't throw")
   - Include attribution comment:
     ```
     # Regression: [bug description]
     # Found by QA Phase 06, Run {RUN_ID}
     ```
3. Run the test — it should PASS if you also fixed the bug, or FAIL if
   the bug is being logged for Phase 07-08 to fix.

### 7. Health Score

Compute a health score using this rubric:

**Per-category scoring** (each starts at 100, deduct per finding):
- Critical issue: -25
- High/Major issue: -15
- Medium issue: -8
- Minor/Low issue: -3
- Minimum 0 per category

**Category weights:**
| Category | Weight |
|----------|--------|
| Console errors | 15% |
| Broken links | 10% |
| Visual issues | 10% |
| Functional bugs | 20% |
| UX problems | 15% |
| Performance | 10% |
| Content issues | 5% |
| Accessibility | 15% |

**Final score:** `score = Σ (category_score × weight)`

## Bug Classification

For each bug found:
- **Critical:** App crashes, data loss, security hole, success criteria fails
- **Major/High:** Feature doesn't work as expected but app doesn't crash
- **Medium:** Degraded experience, workaround exists
- **Minor/Low:** Visual glitch, typo, rough edge
