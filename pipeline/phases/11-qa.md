## Your Task

Systematically test the MVP from a user's perspective. Find every bug.
Be the most demanding customer imaginable. Your goal is not to confirm it
works — it is to find every way it can fail.

## QA Process

### 1. Smoke Test

Can the app start without errors?

```bash
$B goto "$APP_URL"
$B console --errors
$B snapshot -i -a -o "{PHASE_ARTIFACTS}/screenshots/smoke-initial.png"
```

Check:
- Does the main page render without a blank screen?
- Are there JavaScript errors in the console?
- Do critical assets (CSS, JS) load?

If the app won't start or renders a blank page, that is a **Critical** bug.
Log it with exact console errors and stop testing — there is nothing to test.

### 2. Success Criteria Verification

Read the success criteria from `{PHASE_ARTIFACTS}/phase-01-plan-ceo.md`.
Extract the `## Success Criteria` section. Test each criterion literally:

For each criterion:
1. Perform the exact user action described.
2. Observe what happens.
3. Record PASS or FAIL.
4. If FAIL: describe precisely what happened vs. what should have happened.
5. Take a screenshot as evidence.

```bash
$B screenshot "{PHASE_ARTIFACTS}/screenshots/criterion-{N}-{name}.png"
```

A failed success criterion is always at least **Major** severity.

### 3. User Flow Testing

For each distinct user flow in the product spec, execute it end to end:

1. Start from the entry point (homepage, login screen, etc.)
2. Follow the path a real user would take
3. Complete the flow or identify where it breaks
4. Screenshot each meaningful state

```bash
$B goto "$APP_URL"
# Example for a form-based flow:
$B fill "[selector]" "test value"
$B click "[submit selector]"
$B screenshot "{PHASE_ARTIFACTS}/screenshots/flow-{name}-step-{N}.png"
$B snapshot -D  # diff to see what changed
```

### 4. Edge Case Testing

For each user-facing feature, test the unhappy paths:

**Empty / missing input:**
- Submit a form with all fields blank
- Navigate to a detail page for a non-existent item (e.g., `/items/999`)
- What does the user see?

**Invalid input:**
- Enter text where a number is expected
- Enter a date in the past where future is required
- Enter an email without an `@` symbol

**Boundary values:**
- The maximum allowed string length (try length+1)
- Zero or negative numbers where positive is expected
- Special characters: `<script>alert(1)</script>`, `'; DROP TABLE`, `\n`, emoji

**Rapid / repeated actions:**
- Double-click a submit button
- Click a toggle state rapidly 5 times in a row
- Submit the same form twice before the first response returns

**Browser navigation (web apps):**
- Press the back button mid-flow — does state restore correctly?
- Refresh the page mid-flow — is data lost unexpectedly?
- Bookmark a deep link — does it load the correct state?

### 5. Visual Inspection

```bash
$B goto "$APP_URL"
$B snapshot -a -o "{PHASE_ARTIFACTS}/screenshots/visual-annotated.png"
$B screenshot "{PHASE_ARTIFACTS}/screenshots/visual-desktop.png"
```

Check:
- Are elements overlapping or clipping?
- Is text readable? No low-contrast text against similar backgrounds?
- Does it look broken at different window sizes?

**Mobile viewport:**
```bash
$B viewport 375 812
$B screenshot "{PHASE_ARTIFACTS}/screenshots/visual-mobile.png"
$B viewport 1280 720
```

**Tablet viewport:**
```bash
$B viewport 768 1024
$B screenshot "{PHASE_ARTIFACTS}/screenshots/visual-tablet.png"
$B viewport 1280 720
```

### 6. Test Suite Verification

Run the test suite and record results:
```bash
cd output && npm test 2>&1 || python3 -m pytest -v 2>&1 || echo "No test runner"
```

A failing test is a bug. Do not skip failing tests — they must appear in
the bug list.

### 7. Regression Test Generation

For every **new** bug found during this QA cycle that is NOT already covered
by an existing test:

1. Write a regression test that reproduces the exact scenario.
2. The test MUST:
   - Set up the precondition that triggers the bug
   - Perform the action that exposed it
   - Assert the correct behavior (not just "it renders" or "it doesn't throw")
   - Include an attribution comment:
     ```
     # Regression: [bug description]
     # Found by QA Phase 11, Run {RUN_ID}
     ```
3. Place regression tests in `output/tests/regression/`.
4. Run the regression test — it should FAIL without the fix (proving it
   captures the real bug) and will be fixed in Phase 11b.

### 8. Health Score

Compute a health score using this rubric:

**Per-category scoring** (each starts at 100, deduct per finding):
- Critical issue: -25
- High/Major issue: -15
- Medium issue: -8
- Minor/Low issue: -3
- Minimum 0 per category

**Category weights:**
| Category         | Weight |
|------------------|--------|
| Console errors   | 15%    |
| Broken links     | 10%    |
| Visual issues    | 10%    |
| Functional bugs  | 20%    |
| UX problems      | 15%    |
| Performance      | 10%    |
| Content issues   | 5%     |
| Accessibility    | 15%    |

**Final score:** `score = Σ (category_score × weight)`

## Bug Classification

- **Critical:** App crashes, data loss, security hole, success criterion fails,
  blank page on load
- **Major/High:** Feature doesn't work as expected but app doesn't crash
- **Medium:** Degraded experience, workaround exists
- **Minor/Low:** Visual glitch, typo, rough edge
